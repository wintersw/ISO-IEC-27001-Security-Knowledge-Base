# Service Design Security Checklist

- [ ] service owner assigned
- [ ] data classification defined
- [ ] risk assessment completed
- [ ] access model defined
- [ ] logging requirements defined
- [ ] backup/recovery requirements defined
- [ ] supplier dependencies reviewed
- [ ] security testing planned
- [ ] incident escalation defined

## Evidence

Retain the completed checklist with reviewer, date, decisions, and follow-up actions.

## How to use this checklist

Use this checklist for **Service Design Security**. Assign a reviewer, scope and deadline, and evidence for each answer. Failed items require an owned action, exception, or risk decision; a checked box alone is not evidence.

## Practical example

Before approving **Service Design Security**, the reviewer checks an authoritative source, assigns a failed item for remediation, and signs off only after resolution or formal acceptance is evidenced.

## ISO requirement, implementation guidance, and best practice

This checklist is guidance, not prescribed ISO wording. Formal requirements depend on the scoped outcome supported by **Service Design Security**. Evidence retention and independent review of high-risk failures are good practice.
