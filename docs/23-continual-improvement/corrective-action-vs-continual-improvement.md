# Corrective Action vs Continual Improvement

These concepts are related but not identical.

## Correction

Correction fixes the immediate detected problem.

Example:

> Disable a terminated user's account that was left active.

## Corrective action

Corrective action removes the cause of a nonconformity to prevent recurrence.

Example:

> Update the offboarding workflow, integrate HR termination notifications, and verify all application owners receive termination tasks.

## Continual improvement

Continual improvement enhances the information security management system (ISMS) even when there is no nonconformity.

Example:

> Automate access-review evidence collection to reduce manual effort and improve assurance quality.

## Decision guide

| Situation | Use |
|---|---|
| One issue found and fixed immediately | Correction |
| Requirement was not met and could recur | Corrective action |
| Process works but could be stronger | Continual improvement |
| Risk changed | Risk treatment update |
| Control failed | Control redesign and corrective action |
| Evidence is weak | Evidence improvement |

## Audit expectation

Auditors may ask:

- Was the immediate issue corrected?
- Was root cause analyzed?
- Was corrective action implemented?
- Was effectiveness verified?
- Were similar issues considered?
- Was the risk register or SoA updated if needed?

## Practical example

A recurring control failure is entered into the improvement process. The owner analyzes the cause, implements a proportionate change, and verifies after an agreed period that the failure has not recurred.

## Evidence to retain

Retain records showing both design decisions and actual operation, such as:

- improvement source and prioritization decision
- root-cause analysis
- implemented action
- effectiveness review and closure approval
