# Security Process Documentation

The security process should be documented enough to be understandable, repeatable, auditable, and maintainable.

## What should be documented?

- security policy and strategy
- security concept
- roles and responsibilities
- risk methodology
- risk assessment results
- selected controls
- management decisions
- incident investigation results
- internal audit results
- corrective actions
- exceptions
- review results

## Document control requirements

Documentation should be:

- current
- versioned
- approved where necessary
- protected according to sensitivity
- accessible to authorized users
- archived when replaced
- traceable to decisions and evidence

## Current and previous versions

Current versions must be quickly accessible. Previous versions should be retained where they provide evidence of historical decisions or audit periods.

## Best practices

- Use a document register.
- Define owner and review frequency.
- Archive superseded versions centrally.
- Avoid uncontrolled local copies.
- Protect sensitive information security management system (ISMS) documents.
- Link documents to evidence and SoA entries.

## Evidence

- document register
- version history
- archive records
- access control records
- review logs


## Practical example

During a surveillance audit, an auditor asks why a major risk was accepted two years earlier. Because the organization archives superseded ISMS documents centrally, the ISB retrieves the risk assessment version, the management decision record, and the security concept as they stood at the time — demonstrating the historical decision was informed and approved. Without version archival, the organization could only have offered the current documents, which no longer reflected the decision context.
