# Data Security Governance Operating Model

Data security governance turns data protection from isolated controls into repeatable management practice.

## Governance forums

| Forum | Purpose |
|---|---|
| information security management system (ISMS) steering group | risk, resources, priorities, management decisions |
| data governance council | classification, ownership, quality, lifecycle |
| privacy/security review board | high-risk processing and design reviews |
| architecture review | platform, cloud, artificial intelligence (AI), integration, and data-flow design |
| incident review | breach lessons learned and control updates |

## Decision rights

Define who approves:

- new sensitive data collection
- data sharing
- data exports
- access to restricted data
- retention exceptions
- production data in test environments
- high-risk analytics
- AI training data use
- cross-border or third-party processing

## Governance outputs

- data risk register
- data classification register
- data flow maps
- exceptions register
- data security metrics
- management review inputs
- improvement backlog

## Typical evidence

- charters and meeting minutes for data governance forums
- documented decision rights for sensitive data collection, sharing, and exports
- data risk register, classification register, and exceptions register extracts
- data security metrics reported to management review
- improvement backlog items traced to governance decisions

## Practical example

A marketing team wants to export customer emails to a new campaign tool. Under the operating model, the request goes to the data governance council, which checks classification and decision rights, requires a privacy review because personal data leaves the core platform, and records the approval and conditions in the exceptions register — turning a previously informal email approval into a traceable governance decision.
